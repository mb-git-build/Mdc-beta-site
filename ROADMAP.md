# ROADMAP.md

## Purpose

This document is the operational execution guide for long-term development of `modulardatacenters.ai`.

It is not a marketing roadmap.
It is not a public-facing status page.
It is not a pitch deck summary.

It is an internal mission framework for sustained execution.

Together:
- `PROJECT_CHARTER.md` defines philosophy, identity, and doctrine
- `ONTOLOGY_SPEC.md` defines structure, taxonomy, and relationships
- `ROADMAP.md` defines execution priorities, workstreams, boundaries, and autonomous direction

Before major implementation passes, review all three.

---

## 1. Current Project State

The project has crossed an important transition point.

It is no longer primarily a small directory website project.
It is becoming a **typed infrastructure intelligence graph** focused on the physical AI compute ecosystem.

Current state:
- repo is self-contained
- data layer is locally stable
- deployment pipeline is functioning
- architecture is stable enough for incremental growth
- ontology direction has been established
- project doctrine is now documented in-repo
- the next major value layer is ecosystem expansion + utility refinement

### Current interpretation
The frontend is now the presentation layer.
The enduring product asset is:
- ontology
- taxonomy
- metadata
- typed relationships
- discovery pathways
- ecosystem intelligence

## 7. Search / Discovery Goals

Discovery quality is one of the central product value layers.

### Near-term goals
- improve company search relevance
- improve category-level browsing paths
- improve adjacent category navigation
- improve related company visibility
- improve filtering by role, deployment model, and focus area
- improve scanability of browse pages with a larger dataset

### Discovery principle
The system should help a user move from:
- one company
- one category
- one deployment problem

to:
- adjacent systems
- relevant supplier clusters
- comparable vendors
- neighboring infrastructure layers
- overlooked but relevant providers

---

## 8. Ingestion Pipeline Goals

Long-term scale requires better ingestion discipline.

### Near-term goals
- standardize vendor metadata enrichment
- standardize category assignment logic
- standardize subcategory assignment logic
- improve consistency of tags and typed fields
- reduce ad hoc record shapes
- create repeatable enrichment patterns for new vendors

### Mid-term goals
- create more explicit ingestion workflows for adding batches of companies
- build repeatable review logic for metadata quality
- make relationship enrichment less manual over time

### Ingestion doctrine
Do not hardcode future growth one entity at a time forever.
Build repeatable structures that support:
- hundreds
- thousands
- eventually much larger entity sets

without adding unnecessary platform complexity too early.

## 11. Current Bottlenecks

### Strategic bottlenecks
- coverage is still too shallow relative to the long-term vision
- taxonomy is better, but not yet deep enough across all critical layers
- metadata consistency still needs tightening
- related-company logic is early-stage
- category adjacency can become much stronger
- thin pages still exist in parts of the system
- search/filter systems are not yet rich enough for a graph-like discovery experience

### Operational bottlenecks
- ingestion is still too manually intensive
- structured relationship coverage is still partial
- some page usefulness still depends too much on prose rather than structured data

## 13. Explicit Boundaries

### Do not do without explicit request or strong necessity
- major architecture rewrites
- framework migrations
- drastic UX redesigns
- visual reinvention passes
- speculative backend pivots
- unnecessary abstraction layers
- decorative brand overhauls
- platformization for its own sake

### Escalate before acting when work would imply
- replacing major architectural assumptions
- introducing significant operational complexity
- creating a new product category altogether
- changing the project from graph-oriented intelligence back into generic marketing surface work

Do not spend primary energy on:
- homepage redesign churn
- flashy AI visual tropes
- cyberpunk/futuristic styling
- generic startup branding exercises
- speculative dashboards
- enterprise-software theater
- editorial bloat without structured intelligence value
- architecture novelty for its own sake

## 15. Working Mission

The working mission now is:

**expand and refine the structured map of the physical AI compute infrastructure ecosystem**

through:
- deeper ontology
- stronger taxonomy
- richer metadata
- better relationships
- broader coverage
- stronger discovery
- incremental, utility-driven UI refinement

The correct default move is usually:
- improve the map
- improve the structure
- improve the discovery

not:
- reinvent the machine

---

## 16. Execution Heuristic

When choosing what to work on next, prefer tasks that most improve:
1. ecosystem intelligence
2. ontology consistency
3. discoverability
4. long-tail coverage
5. structured utility
6. page usefulness through structured data

If a task does not improve at least one of those in a meaningful way, it is probably not the right next move.
